Claude Code · Shared Harness

How the claude-harness setup works

A project-neutral plugin marketplace that every repository reuses, so the way Claude Code works — skills, subagents, and session automation — is identical across all projects, while each project keeps only its stack-specific bits locally.

github.com/JDSavvy/claude-harness · Claude-Code-only · prepared 2026-06-15

1 The one-sentence model

GitHub is the source of truth. Claude Code clones it once per machine into ~/.claude/plugins/. Every project on that machine uses that single local copy — there is no per-project copy.

2 The flow

3 Is it local? Yes.

After you add the marketplace and install the plugin, Claude Code keeps a local clone and an extracted, versioned cache under your home directory. Projects do not fetch from GitHub on every run — they read the local copy.

Path What it is
~/.claude/plugins/marketplaces/claude-harness/ git clone of the marketplace repo
~/.claude/plugins/cache/claude-harness/harness/<version>/ the extracted plugin used at runtime (skills, agents, hooks)
~/.claude/plugins/installed_plugins.json installed version + git commit SHA
~/.claude/plugins/known_marketplaces.json registered marketplaces + last refresh

4 Integrate into a NEW project — two ways

A) Machine-wide, once (interactive)

/plugin marketplace add https://github.com/JDSavvy/claude-harness.git
/plugin install harness@claude-harness

Now every project on that machine has /create-issue, the agents, and the hooks.

B) Repo-committed bootstrap (recommended)

Commit this to the project's .claude/settings.json:

{
  "extraKnownMarketplaces": {
    "claude-harness": {
      "source": { "source": "git", "url": "https://github.com/JDSavvy/claude-harness.git" },
      "autoUpdate": true
    }
  },
  "enabledPlugins": { "harness@claude-harness": true }
}
When someone trusts the repo folder, Claude Code prompts them to install it — it is not a silent auto-install (Claude Code security design). This is what a consuming project commits, so a fresh clone is one click away from the full harness.

5 Updates

  • Automatic: autoUpdate: true → Claude Code refreshes the local clone + plugin at session startup.
  • Manual: /plugin marketplace update claude-harness.
  • Notification: the harness-update-check.sh SessionStart hook does a live ls-remote (throttled ~daily) and tells you when your local copy is behind GitHub. It never auto-pulls a globally-shared plugin mid-session.

6 Shared vs. per-repo

Shared (the plugin, one copy) Per-repo (committed in each project)
/create-issue, /finish-pr
generic code-reviewer, test-runner
SessionStart hooks (git-sync, update-check, anti-hallucination) 		CLAUDE.md (commands, conventions)
app skills + stack-specific subagents
stack hooks (formatter, i18n, guards)
local quality gate (e.g. Lefthook pre-push)
Why two layers: the handling is identical everywhere (one place to improve it), while each project keeps only what is truly stack-specific. Local items with the same name (e.g. a project's code-reviewer) shadow the generic ones — local wins.

7 The @claude GitHub Action

In CI the action loads the same marketplace independently via workflow inputs — it clones the public repo fresh inside the runner (no local cache there): 

with:
  claude_code_oauth_token: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}
  plugin_marketplaces: https://github.com/JDSavvy/claude-harness.git
  plugins: harness@claude-harness

The repo is public, so the default GITHUB_TOKEN can clone it — no PAT needed.
